🔨 What We Built

An intelligent IT Helpdesk triage agent that ingests inbound support requests, classifies them into one of seven routing queues (SECURITY, NETWORKING, IDENTITY, HARDWARE, SOFTWARE, AUTO_RESOLVE, ESCALATION), assigns a P1–P4 priority, and decides the appropriate action — auto-resolve, route, or escalate — all via the Claude API on AWS Bedrock.

The agent runs a full agentic loop: it calls two tools (knowledge_lookup and check_status) to enrich its context before classifying. It enforces a hard guardrail layer before touching Claude (keyword triggers, unknown sender domains) and applies post-loop mandate rules (confidence threshold, P1/P2 auto-escalation, SECURITY lockout).

The eval harness (12 test cases) covers happy paths, edge cases, and adversarial attempts to bypass guardrails. All logic is real and runnable; mock data is provided for the FAQ knowledge base and system-of-record tickets.

⚡ Key Decisions

  • Pre-Claude guardrail layer — keyword and domain checks run before any API call, so malicious or out-of-mandate requests never reach the model. Keeps cost and attack surface low.
  • Confidence threshold enforced post-loop — Claude returns a confidence score; the Python layer, not the model, decides whether to escalate. This makes the mandate auditable in code, not hidden in a prompt.
  • Mock data in JSON files — no database dependency for the competition; the tool layer reads flat files. Replacing them with real API calls requires changing one function each.
  • Max tool iteration cap at 5 — prevents runaway loops; if exceeded the ticket auto-escalates rather than hanging or burning tokens.

🤖 How We Used Claude Code

Claude Code scaffolded the entire project from the CLAUDE.md spec — all five challenges were generated in a single session with explicit task tracking via the todo list. The CLAUDE.md acted as a living contract: domain terminology, queue names, priority SLAs, and tool schemas were defined once and referenced consistently across every generated file.

The most effective workflow was writing the mandate and architecture docs first, then having Claude use those as context when generating the agent code. This meant the routing logic in triage_agent.py exactly matched the mandate table without needing a correction pass.
